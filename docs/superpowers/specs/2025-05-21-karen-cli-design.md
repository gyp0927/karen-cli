# karen-cli 设计文档

## 1. 项目概述

karen-cli 是一个类 Claude Code 的通用编码助手 CLI，采用 TypeScript/Node.js 技术栈，支持多模型切换。

核心理念：**模型负责决策，Harness 负责执行**。

**迭代策略**：
- 第一阶段（MVP / 7.1）：完成核心循环 + 基本工具 + 多模型
- 第一阶段完成后，经用户确认，自动进入第二阶段（7.2）：依次实现 Memory、Task Graph、Skill Loader、Hooks、上下文压缩、子 Agent 委派
- **每个功能完成后必须测试，且全部已有功能回归测试通过后才能继续下一步**

**核心理念**：
- **Agent Loop** 是稳定内核，一旦实现就不重写
- 外围机制（工具、权限、记忆等）以模块方式挂载到 Loop 上
- 每个机制有单一职责，通过明确接口与核心通信

## 3. 核心组件

### 3.1 Agent Core Loop (`src/core/loop.ts`)
- 稳定内核，整个系统的引擎
- 职责：组装上下文 → 调用模型 → 解析工具调用 → 调度执行 → 收集结果 → 决定是否继续循环
- 关键：不直接关心具体工具是什么，只通过 Tool Registry 调度

### 3.2 REPL / 命令解析器 (`src/cli/repl.ts`)
- 交互式终端界面，处理用户输入
- 支持命令历史、多行输入、退出命令（`/exit`, `/quit`）
- 特殊命令：`/model` 切换模型，`/tools` 查看可用工具，`/tasks` 查看任务状态

### 3.3 Tool Registry (`src/tools/registry.ts`)
- 统一管理所有工具（Read、Write、Edit、Bash、Grep、Glob、Agent 等）
- 每个工具实现统一接口：`{ name, description, parameters, execute }`
- 核心循环通过 registry 查找并调用工具，无需硬编码

### 3.4 Model Provider Layer (`src/providers/`)
- 统一接口 `IProvider`，屏蔽不同模型的差异
- 初始实现：`AnthropicProvider`、`OpenAIProvider`
- 可扩展：`OllamaProvider`、`AzureProvider` 等
- 支持运行时切换：`/model claude` 或 `/model gpt4`

### 3.5 Memory System (`src/memory/`)
- 持久化存储：`.karen/memory/` 目录下的 Markdown 文件
- 类型：user（用户信息）、feedback（反馈）、project（项目上下文）、reference（外部引用）
- 按消息索引自动加载相关记忆到上下文

### 3.6 Task Graph (`src/tasks/`)
- 追踪多步骤任务的执行状态
- 支持任务依赖（A 完成前 B 不能开始）
- CLI 可查看：`/tasks` 列出当前任务状态

### 3.7 Skill Loader (`src/skills/`)
- 从 `src/skills/` 或外部目录加载自定义技能
- 技能定义：name、description、trigger conditions、prompt template
- 模型可主动调用技能，类似工具调用

### 3.8 Permission & Hook System (`src/permissions/`, `src/hooks/`)
- 权限：每次执行敏感操作（Bash、Write）前询问用户
- Hooks：生命周期钩子（pre-message、post-tool、pre-exit 等）
- 可配置：`.karen/settings.json` 设置默认权限策略

## 4. 数据流

### 4.1 单次对话循环

```
用户输入 ──▶ REPL ──▶ Agent Loop
                           │
                           ▼
              ┌──────────────────────┐
              │ 1. 组装上下文消息    │
              │    (系统提示 + 历史   │
              │     + 记忆 + 当前输入)│
              └──────────────────────┘
                           │
                           ▼
              ┌──────────────────────┐
              │ 2. 调用 Model        │
              │    Provider          │
              └──────────────────────┘
                           │
                           ▼
              ┌──────────────────────┐
              │ 3. 解析模型响应      │
              │    (文本 / 工具调用) │
              └──────────────────────┘
                           │
              ┌────────────┴────────────┐
              ▼                          ▼
          [纯文本回复]              [工具调用]
              │                          │
              ▼                          ▼
          直接输出用户          Tool Registry 查找
                                       │
                                       ▼
                              权限检查（敏感操作？）
                                       │
                              ┌────────┴────────┐
                              ▼                 ▼
                            允许              拒绝
                              │                 │
                              ▼                 ▼
                          执行工具          返回错误信息
                              │            给模型
                              ▼
                          收集结果
                              │
                              ▼
                         返回 Loop
                         (继续循环直到
                          模型不再调用工具)
```

### 4.2 工具调用详细流

当模型返回工具调用时：
1. **解析**：提取工具名和参数
2. **查找**：Tool Registry 中匹配工具
3. **权限检查**：如果是 `Bash`、`Write` 等敏感操作，询问用户确认
4. **执行**：调用工具的 `execute` 方法
5. **结果格式化**：将执行结果转为消息，送回模型
6. **循环判断**：模型根据结果决定继续调用工具还是回复用户

### 4.3 记忆加载流

```
Agent Loop 组装上下文时
         │
         ▼
   Memory System 检索
   (根据当前输入关键词
    匹配相关记忆文件)
         │
         ▼
   加载匹配的记忆内容
   注入到系统提示或
   user message 中
         │
         ▼
   模型获得上下文增强
```

## 5. 上下文压缩机制

### 5.1 触发条件

| 指标 | 阈值 | 动作 |
|------|------|---------|
| 当前上下文 Token 数 | > 80% 模型上下文窗口 | 触发压缩 |
| 连续对话轮次 | > 50 轮 | 强制压缩 |
| 单次工具调用结果 | > 20% 窗口 | 截断 + 摘要 |

### 5.2 压缩策略（分层）

```
完整上下文
    │
    ▼
┌────────────────────────────────────────┐
│ 1. 系统提示（System Prompt）            │
│    └── 始终保留，不可压缩               │
├────────────────────────────────────────┤
│ 2. 活跃记忆（Memory）                   │
│    └── 保留，但只保留高相关度条目       │
├────────────────────────────────────────┤
│ 3. 当前任务上下文（Task Context）       │
│    └── 保留，但压缩已完成任务的细节     │
├────────────────────────────────────────┤
│ 4. 历史对话（Messages）                 │
│    └── 【压缩目标】                     │
│        - 早期轮次 → 提取摘要            │
│        - 中间工具调用/结果 → 丢弃详情   │
│        - 保留最近 N 轮完整对话          │
└────────────────────────────────────────┘
```

### 5.3 压缩具体方式

**方式 A：对话摘要（用于历史消息）**
- 将早期多轮对话交给模型生成一段摘要
- 摘要替代原始消息进入上下文
- 摘要保留：用户意图、关键决策、已执行的操作结果

**方式 B：工具结果截断（用于大输出）**
- Bash/Grep 等输出超长时，保留前 100 行 + 后 50 行
- 中间用 `[... N lines truncated ...]` 替代
- 如果模型需要完整内容，可再次调用工具

**方式 C：滑动窗口（最简单）**
- 直接丢弃最旧的消息，只保留最近 N 轮
- 优点：简单、确定性强
- 缺点：可能丢失关键上下文
- 可作为兜底策略

### 5.4 与记忆系统的区别

| | 上下文压缩 | 记忆系统 |
|--|-----------|---------|
| **时机** | 运行时自动触发 | 显式保存（成功/失败时） |
| **内容** | 临时对话历史 | 持久化的关键信息 |
| **目的** | 解决 Token 超限 | 跨会话保持知识 |
| **存储** | 内存中 | `.karen/memory/` 文件 |

**协作方式**：压缩前，提取的关键信息（如用户偏好、项目决策）可自动写入记忆，确保不丢失。

## 6. 错误处理

### 6.1 错误分类

| 层级 | 错误类型 | 处理方式 |
|------|---------|---------|
| **Provider** | API 调用失败（超时、限流、密钥无效） | 重试（指数退避）→ 切换 Provider（可选）→ 报错给用户 |
| **Tool** | 工具执行失败（文件不存在、命令报错） | 捕获错误 → 格式化错误信息 → 送回模型，让模型决定下一步 |
| **Permission** | 用户拒绝敏感操作 | 返回拒绝信息给模型，模型可调整策略或询问用户 |
| **Loop** | 无限循环、上下文超限 | 最大轮次限制（如 25 轮）、Token 预警、强制截断 |
| **Parse** | 模型输出格式异常（非标准 JSON） | 尝试容错解析 → 失败则请求模型重新格式化 |

### 6.2 核心原则

- **不静默失败**：所有错误都需明确反馈（给用户或给模型）
- **模型可恢复**：工具执行错误要送回模型，让它有机会修正
- **优雅降级**：API 失败时提供降级路径（如切换模型或离线模式）

## 10. 开发原则

1. **测试驱动**：每个功能完成后必须测试，全部已有功能回归通过才能继续下一步
2. **增量叠加**：不重写核心循环，新能力以模块方式挂载
3. **边界清晰**：每个模块单一职责，通过接口通信
4. **错误透明**：不静默失败，所有错误明确反馈
